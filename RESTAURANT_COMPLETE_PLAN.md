## 1. Why Three Stages

**Stage A** — design once. Research, positioning, page architecture, visual direction specific to restaurants.

**Stage B** — build once. A production-ready master template (The Meridian) that no individual client ever sees directly.

**Pipeline B** — scale forever. Each new restaurant client gets a fully customized site in under 2 minutes via automated 7-step pipeline.

```
Stage A (Design)  →  Stage B (Build)  →  Pipeline B (Scale)
   1-2 weeks            5 weeks          <2 min per client
   One time             One time          Repeatable
```

Without this: every new client = hours of manual admin editing + risk of template contamination.
With this: every new client = intake form submission → automated pipeline → live site.

## 3. A1: Industry Research

*Full detail: `a1_industry_brief.md`*

**Discovery patterns by sub-type:**
- Fine Dining: Google → photography first → press validation. Mobile = 68–74% of traffic.
- Cafe/Brunch: Google Maps "near me". "Open Now" badge above fold = #1 conversion moment.
- Bar/Lounge: Late-night hours + reservation policy must be instantly clear.

**Top mobile failures:** PDF menu · Broken OT/Resy widget on iOS Safari · Hero crops wrong on portrait · No `tel:` link · Page weight >1.5MB.

**Multilingual:** Chinese guests rely on dish photos + Chinese menu names. Spanish guests: higher phone reservation rate, community word-of-mouth.

**Reservation strategy:** Fine Dining → Resy · Cafe/Brunch → custom or none · Bar/Lounge → OpenTable or custom.

**10 premium design differentiators:** editorial typography · art-directed photography · white space as luxury · 2–3 color maximum · considered hover states · chef as character · seasonal awareness · mobile nav with large targets · cross-touchpoint consistency · purposeful negative space.

## 10. Stage B Overview

**Duration:** 5–6 weeks · **Tool:** Cursor AI (claude-sonnet-4-6 Agent)
**Baseline:** drhuangclinic.com multi-tenant platform (fork + strip)

### Core Methodology

**Contract-First:** JSON schema + variants + admin form fields defined BEFORE coding each page.

**Admin Done-Gate (every page):**
1. BUILD — renders from DB, variant-aware, theme tokens only
2. WIRE — admin Form mode correct, JSON mode correct, variant dropdown present
3. VERIFY — roundtrip: Form edit → save → frontend updated; variant switch → layout changes; JSON edit → Form syncs

### Architecture Principles

1. DB-First — frontend reads from Supabase `content_entries`; local JSON = dev fallback only
2. Theme Tokens — no hardcoded hex anywhere; all via CSS variables
3. Host-Based Resolution — middleware reads hostname → `site_domains` → `SiteContext`
4. Site-Scoped Isolation — all content scoped by `site_id`; RLS enforced
5. Media via Supabase Storage — all images via CDN; no local filesystem in production
6. Feature Flags — 20 toggleable features; no code changes per client
7. Multilingual Native — `LocalizedString` everywhere; never auto-translate

## 17. Pipeline B Overview

> **Full detail: `RESTAURANT_CLIENT_ONBOARDING_MASTER_PLAN.md`**

### The Flow

```
Restaurant Intake Form (JSON)
         ↓
   Pipeline B (7 steps)
         ↓
Customized Client Site
(~$0.13 · ~30-50 seconds)
```

### What Changes Per Client

| What | How |
|---|---|
| Business name, address, phone, email, domain | Deterministic replacement (O4) |
| Brand colors, fonts | Variant selection + shade generation (O2) |
| Menu types offered | Pruning — remove unsupported types (O3) |
| Hero copy, about story, bios, testimonials | AI generation via Claude API (O5) |
| SEO titles + descriptions per page | AI generation via Claude API (O5) |
| Languages supported | Locale cleanup (O6) |
| Reservation provider + ID | Structural rebuild (O4) |
| Feature flags | `sites` table update (O4) |

### What Stays (cloned)
Page structure · React components · UI system · Admin dashboard · API endpoints · Layout files.

## 18. 7-Step Pipeline (O1–O7)

| Step | Name | Duration | What It Does |
|---|---|---|---|
| O1 | Clone | ~5s | Copy all Meridian `content_entries` to new `site_id`; register domains |
| O2 | Brand | <1s | Apply `brand-variants-restaurant.json` variant; override colors/fonts from intake |
| O3 | Prune | ~3s | Delete disabled menu types from 5 file types |
| O4 | Replace | ~5s | Deep string replacement (17 pairs); rebuild hours, team, social, reservation config |
| O5 | AI Content | ~20s | Claude API call 1: hero/bios/testimonials · Call 2: all-page SEO |
| O6 | Cleanup | <1s | Delete `content_entries` rows for unsupported locales |
| O7 | Verify | <1s | Required paths exist · no contamination · menu count matches intake |

**Total:** ~30–50 seconds with AI · ~15 seconds skip-AI mode.

**Execution methods:**
- Admin UI wizard at `/admin/onboarding` (SSE progress streaming)
- CLI: `node scripts/onboard-client.mjs {client-id}`

## 24. Multi-Tenant Architecture

```typescript
// middleware.ts — host-based site resolution
export function middleware(request: NextRequest) {
  const host    = request.headers.get('host')         // "labellacita.com"
  const site    = resolveSiteFromHost(host)           // queries site_domains table
  const locale  = detectLocale(request, site)         // cookie → accept-language → default

  const headers = new Headers(request.headers)
  headers.set('x-site-id', site.id)
  headers.set('x-locale', locale)

  return NextResponse.rewrite(
    new URL(`/${locale}${request.nextUrl.pathname}`, request.url),
    { headers }
  )
}
```

All `content_entries` rows scoped by `site_id`. RLS enforced. Admin in one site cannot see another.

**Dev local domains:** Each client gets `{alias}.local` → `/etc/hosts` → `127.0.0.1`. Pipeline O1 auto-registers in `site_domains`. Only manual step: `sudo sh -c 'echo "127.0.0.1 {alias}.local" >> /etc/hosts'`.

**Scaling:** 1–100 sites: single Supabase project · 100–500: monitor performance · 500+: shard by site_id.

## 29. Anti-Patterns & Lessons Learned

| Anti-Pattern | Fix |
|---|---|
| Menu as PDF | HTML only, DB-driven, admin-editable |
| Hours hardcoded | Hours from `site.json` via DB always |
| Single reservation widget | 3 variants required (OT / Resy / custom) |
| Photography as local files | All in Supabase Storage + CDN |
| English-only image alt | All alt text = LocalizedString |
| Seasonal menu as separate site | Seasonal flag + badge on existing items |
| Static events calendar | Events = Supabase collection |
| Collection editors built in Phase 3 | Build with every Phase 1–2 page (menus = #1 content update frequency) |
| Theme built for one variant | All 4 variants in Phase 0 |
| Manual client onboarding | Pipeline B — admin "New Site" = O1 only; 6 more steps needed |
| Template strings in production | O7 catches this — never skip O7 |
| Skip AI for real clients | $0.13 is negligible; template hero destroys trust |
| Partial-match in replacement | Sort replacement pairs by length descending |
| Share Supabase project across industries | Different schemas cause migration conflicts |
