## Track 1: Content Publishing — First 6 Months

### Month 1–2: Launch Burst (8–10 Posts)

The most important SEO action after launch is publishing a strong initial content foundation quickly. Google treats newly-launched sites with more scrutiny — a burst of quality content signals that the site is legitimate and active.

**Month 1 Target: 5 posts published**

Priority order — start with highest search volume, clearest intent:

```
POST 1 (Week 1) — Category: Seasonal Guide
Slug: /en/blog/best-dishes-for-first-time-visitors
Title: "The {N} Dishes Every First-Time Visitor to {Restaurant} Should Try"
Why first: High search intent ("best dishes at X") + immediately useful
            for reservations coming through new site
Target keyword: "{restaurant name} menu recommendations"
Content brief:
  - Opening: Why certain dishes define what the restaurant is about
  - Feature each dish: 1 paragraph per dish, what makes it special,
    what to pair it with
  - Chef quote on the signature dish
  - Close: "Reserve your table to try them →" CTA
  Length: 800–1200 words
  SEO: title includes restaurant name + menu/dishes
  Image: 1 feature photo per dish (at least 3 food photos)

POST 2 (Week 1) — Category: Chef's Perspective
Slug: /en/blog/our-philosophy-seasonal-local
Title: "Why We Only Use Seasonal, Local Ingredients"
Why second: Establishes brand voice + targets "farm to table [city]" queries
Target keyword: "farm to table [city]", "local sourcing restaurant [city]"
Content brief:
  - Chef's sourcing philosophy in their own words
  - Name 2–3 specific local farms or suppliers
  - How sourcing affects what's on the menu
  - Seasonal menu change preview for upcoming season
  Length: 700–1000 words

POST 3 (Week 2) — Category: Seasonal Guide
Slug: /en/blog/guide-to-the-{menu-type}-menu
Title: "A Complete Guide to Our {Dinner/Tasting/Brunch} Menu"
Why third: Ranks for "[restaurant] menu" + "[cuisine] menu [city]"
Content brief:
  - Walk through each menu section
  - Describe 2–3 standout items per section
  - Wine/cocktail pairing suggestions for each section
  - Dietary options (vegetarian, GF) called out specifically
  Length: 1000–1500 words (this will be the longest post — high value)

POST 4 (Week 2) — Category: Events & Announcements
Slug: /en/blog/upcoming-events-[current-month]
Title: "What's On at {Restaurant} in {Month}: Events + Specials"
Why fourth: Topical, date-specific content = easy to rank for
            "[restaurant] events [month]" queries
Content brief:
  - All upcoming events for the month
  - Seasonal specials + limited menu items
  - "Reservation availability update" (how far out they're booking)
  Length: 500–700 words (this is a monthly post — keep it light)

POST 5 (Week 3) — Category: Wine & Spirits
Slug: /en/blog/wine-pairing-guide-beginners
Title: "A Beginner's Guide to Wine Pairing at {Restaurant}"
Why fifth: High search volume, brings in pre-visit research traffic
Target keyword: "wine pairing [cuisine] food", "wine pairing guide"
Content brief:
  - 3 principles of wine pairing explained simply
  - Specific pairing recommendations for signature dishes
  - How to order wine at the restaurant (the process)
  - Sommelier or Bar Director introduction
  Length: 800–1000 words
```

**Month 2 Target: 3–5 more posts**

```
POST 6 — Behind the Scenes: "A Day in Our Kitchen"
```

### Month 3–6: SEO Compounding (2–3 Posts/Month)

After the launch burst, drop to a sustainable 2–3 posts per month. Quality over quantity from here.

**Best-performing post types for restaurant sites:**

```
HIGH RETURN:
  "[Cuisine] Restaurants in [City]" — ranks for competitor queries
  "[Occasion] Restaurants in [City]" — anniversary, birthday, date night
  "Things to Do in [Neighborhood] — Food Guide" — local guide content
  "Wine Pairing with [Specific Dish]" — long-tail, high purchase intent
  "How We Make [Signature Dish]" — kitchen process, unique, hard to copy

MEDIUM RETURN:
  "Behind the Scenes: [Event or Process]"
  "Meet the Team: [Staff Profile]"
  "Why [City]'s Food Scene Is Thriving"

SEASONAL (do every year):
  Valentine's Day preview (publish 3 weeks before Feb 14)
  Holiday Season at [Restaurant] (publish early November)
  Summer menu launch (publish when seasonal menu changes)
  NYE announcement (publish early December)
```

**Blog post SEO checklist (apply to every post):**

```
Before publishing each post, verify:
  [ ] Title: includes primary keyword (not just restaurant name)
  [ ] Title length: 50–65 characters
  [ ] URL slug: short, contains keyword, no stop words
  [ ] Meta description: 150–160 chars, ends with call to action
  [ ] H1: matches title exactly (or very close)
  [ ] H2s: section headers include secondary keywords
  [ ] First paragraph: mentions primary keyword within first 100 words
  [ ] Word count: 700+ for evergreen, 400+ for announcements
  [ ] Internal links: at least 2 links to other pages on the site
      (e.g., link to /menu from a menu post, link to /reservations from any post)
  [ ] External links: 1–2 reputable sources cited (not competitors)
  [ ] Featured image: alt text includes keyword
  [ ] Author: set to team member (builds E-E-A-T signal)
  [ ] Schema.org BlogPosting: auto-generated (from Phase 2C)
  [ ] Category: set correctly (used by category filter)
  [ ] Published date: set to today (or intended publish date)
```

### Pipeline B Scale Timeline

Track new clients by quarter. These are conservative targets — actual pace depends on your sales effort.

**Month 1–2: First 2–3 clients (validation phase)**

Goal: confirm the pipeline works consistently with diverse real inputs. Expect some pipeline bugs to surface from edge cases in real intake data (unusual characters in names, non-standard hours, rare menu types). Fix them as they appear.

```
Success criteria for validation phase:
  [ ] 2–3 clients onboarded end-to-end
  [ ] Each client: contamination scan exits 0
  [ ] Each client: no hardcoded Meridian strings visible
  [ ] Total time per client: < 3 hours (improving toward 2 hours)
  [ ] Zero content rollbacks needed (pipeline output acceptable as-is)
  [ ] At least 1 client using non-default brand variant
  [ ] At least 1 client with locales other than English-only
```

**Month 3–4: Scale to 5–10 clients**

Pipeline is proven. Focus shifts to intake efficiency and reducing human post-pipeline time.

```
Optimizations to implement:
  [ ] Menu import: build a CSV/PDF menu importer
      (reduces menu entry from 45 min to 10 min)
      CSV format: name, description, price, category, dietary_flags
      Admin → Menu Editor → Import from CSV
      Script: scripts/import-menu-csv.mjs {client-id} {file.csv}

  [ ] Photo processing: build a bulk photo importer
      Accepts folder of images → uploads all to Supabase Storage
      → creates gallery_items rows with auto-categorization
      Script: scripts/import-gallery.mjs {client-id} {./photos/}

  [ ] Intake form improvements:
      Track which intake fields the AI uses most effectively
      Add any missing fields that clients consistently need
      Improve contentTone descriptions based on AI output quality

  [ ] Admin guide: create a video walkthrough (Loom, 5 min)
      Reduces client support questions significantly
```

**Month 5–6: 10–25 clients, Pipeline B fully autonomous**

At this scale, the goal is reducing your total time per client to under 1 hour (mostly menu entry and photo upload — the irreducible human steps).

```
Platform improvements at this scale:
  [ ] Multi-site admin: operator can switch between client sites
      from a single admin login
      /admin → site switcher dropdown (top of sidebar)
      Shows: all sites this operator has access to
      Click → switches current editing context to that site

  [ ] Client self-serve portal (optional, Phase 5 extension):
      Clients get their own /admin login for their site only
      They can edit content, blog posts, menu items
      They cannot: change site_id, access other clients' data,
        modify theme or structural settings

  [ ] Pipeline B analytics dashboard:
      /admin/pipeline-analytics
      Shows: total sites created, avg onboarding time,
        AI token costs, contamination scan pass rate,
        post-pipeline menu entry time, client-by-client log

  [ ] Automated QA on pipeline output:
      After each pipeline run: auto-run npm run qa:schema + qa:routes
        on the new client site
      FAIL → alert operator before client handoff
      PASS → automated "site ready" email to operator
```

## Phase 5 Never-Do List

Anti-patterns that become more tempting as the client base grows. Never do them.

```
❌ NEVER edit content directly in Supabase for a client
   (always use admin panel — maintains audit trail)

❌ NEVER share the service_role key with clients
   (they get anon key only, if needed at all)

❌ NEVER put client-specific code in the Next.js codebase
   (all client differences live in content_entries + DB)

❌ NEVER run Pipeline B without O7 verification
   (contamination in production = emergency rollback + manual fix)

❌ NEVER skip the intake JSON validation step
   (invalid intake → partial pipeline run → corrupted client site)

❌ NEVER modify the-meridian template content during client work
   (template changes must be deliberate, versioned, and re-tested)

❌ NEVER reuse one client's site_id for a different client
   (always new site_id — content isolation is non-negotiable)

❌ NEVER publish AI content without human review of the hero headline
   (everything else can be AI-drafted, but the first thing visitors read
    must feel authentic to this specific restaurant)

❌ NEVER defer the monthly GSC review
   (index coverage issues left unresolved for 60+ days = months lost)
```
